# AGENTS.md

## 목적

이 파일은 Codex가 어떤 프로젝트에서든 공통으로 따를 수 있는 작업 규칙이다.

목표는 다음과 같다.

- Codex가 스스로 코드 구조를 파악하고 적절한 구현 방식을 선택하도록 한다.
- 작업 범위가 너무 커져서 오류나 토큰 소모가 증가하지 않도록 한다.
- 사용자가 매번 긴 지시문을 작성하지 않아도 되도록 한다.
- 프로젝트별 세부 요구사항은 별도 문서나 사용자의 현재 요청을 기준으로 판단한다.

---

## 기본 작업 원칙

1. 사용자의 요청과 프로젝트의 현재 코드 구조를 먼저 파악한다.
2. 전체 작업을 무리하게 한 번에 구현하지 않는다.
3. 작업 범위가 크거나 여러 기능이 섞여 있으면 먼저 작업 계획을 작성한다.
4. 작업 계획이 필요한 경우 `PLANS.md`를 작성하거나 갱신한다.
5. `PLANS.md` 작성 또는 갱신 전에는 코드를 수정하지 않는다.
6. 사용자가 특정 단계나 작업 범위를 승인하면 해당 범위 안에서 구현한다.
7. 구현 방법은 Codex가 현재 코드 구조에 가장 적합한 방식으로 자율적으로 선택한다.
8. 기존 구조를 존중하되, 요구사항을 만족하기 위해 필요한 범위의 구조 개선은 허용된다.
9. 요청 범위를 벗어난 기능을 임의로 추가하지 않는다.
10. 불필요한 대규모 리팩터링은 하지 않는다.
11. 같은 기능을 중복 구현하지 않는다.
12. 이전 단계에서 완료된 기능을 다시 갈아엎지 않는다.
13. 코드 수정 후에는 변경 내용과 테스트 방법을 요약한다.

---

## 자율 구현 원칙

Codex는 사용자가 세부 구현 방식을 지정하지 않은 경우 다음 원칙에 따라 스스로 판단한다.

- 기존 코드 스타일과 아키텍처를 우선 따른다.
- 이미 존재하는 유틸리티, 설정 저장 방식, 리소스 로딩 방식, 테스트 방식을 우선 재사용한다.
- 중복 코드가 생기지 않도록 공통화가 필요한 부분은 적절히 정리한다.
- 단, 공통화나 구조 개선이 작업 범위를 크게 키우면 먼저 사용자에게 보고한다.
- 성능, 유지보수성, 안정성 중 하나를 명확히 희생해야 하는 경우 사용자에게 선택지를 제시한다.
- 사용자가 “알아서”, “깔끔하게”, “기존 구조에 맞게”라고 요청한 경우에도 요구사항을 벗어나는 기능 추가는 하지 않는다.

---

## 작업 계획 생성 규칙

다음 경우에는 먼저 `PLANS.md`를 작성하거나 갱신한다.

- 요구사항이 여러 개의 기능을 포함하는 경우
- UI, 설정, 저장, 단축키, 파일 처리, 외부 리소스 등 여러 영역이 함께 바뀌는 경우
- 수정 파일이 여러 개가 될 가능성이 높은 경우
- 사용자가 “전체적으로”, “전반적으로”, “개선사항”, “계획”, “단계별” 같은 표현을 사용한 경우
- 이전 작업이 중단되어 이어서 진행해야 하는 경우

`PLANS.md`에는 가능한 한 다음 항목을 포함한다.

- 전체 요구사항 요약
- 단계별 작업 계획
- 각 단계의 목표
- 작업 범위
- 제외 범위
- 예상 수정 파일
- 테스트 방법
- 위험 요소
- 사용자 확인이 필요한 사항

`PLANS.md` 작성 후에는 코드 수정 없이 사용자 승인을 기다린다.

사용자가 승인한 단계 또는 승인한 작업 범위만 구현한다.

---

## 단계 구분 원칙

작업 단계를 나눌 때는 다음 기준을 따른다.

1. 각 단계는 독립적으로 구현 및 테스트 가능해야 한다.
2. 서로 다른 성격의 작업은 가능한 분리한다.
   - 예: UI 변경, 설정 저장, 단축키, 파일 처리, 렌더링, 성능 개선, 버그 수정
3. 기존 기능 수정과 신규 기능 추가는 가능한 분리한다.
4. 영향 범위가 큰 작업은 작은 단계로 나눈다.
5. 한 단계에서 너무 많은 파일을 수정해야 한다면 단계를 더 작게 나눈다.
6. 사용자가 단계 수를 지정하지 않은 경우 Codex가 적절한 단계 수를 제안한다.
7. 단계 수는 고정하지 않되, 너무 많거나 너무 적지 않게 유지한다.

---

## 구현 전 확인 규칙

코드 수정 전에는 다음 내용을 간단히 제안한다.

- 이번 작업에서 수정할 파일
- 구현 방향
- 기존 기능에 미칠 영향
- 제외할 작업 범위
- 예상 위험 요소
- 테스트 방법

사용자가 “진행”, “구현”, “수정해”, “해줘” 등으로 승인하면 코드를 수정한다.

단, 사용자가 명시적으로 바로 수정하라고 요청한 작은 작업은 이 과정을 간소화할 수 있다.

---

## 구현 후 보고 형식

구현 후에는 다음 내용을 간단히 보고한다.

- 수정한 파일 목록
- 주요 변경 내용
- 기존 동작과 달라진 점
- 테스트 방법
- 테스트 결과 또는 확인 필요 사항
- 남은 이슈

보고는 간결하게 작성하되, 사용자가 다음 작업을 판단할 수 있을 만큼 충분히 구체적으로 작성한다.

---

## 토큰 절약 원칙

1. 필요 이상으로 전체 문서나 전체 코드를 반복해서 해석하지 않는다.
2. 이미 작성한 `PLANS.md`가 있으면 우선 그것을 기준으로 진행한다.
3. 이전 단계에서 완료된 기능을 다시 구현하지 않는다.
4. 코드 검색은 현재 작업 범위와 관련된 파일 위주로 수행한다.
5. 사용자가 요청하지 않은 대규모 리팩터링은 하지 않는다.
6. 한 번에 여러 단계를 구현하지 않는다.
7. 오류 수정 시에도 현재 승인된 범위 안에서 해결한다.
8. 긴 설명보다 변경 요약, 영향 범위, 테스트 방법을 우선 보고한다.
9. 동일한 내용을 반복해서 설명하지 않는다.
10. 불확실한 부분은 추측으로 진행하지 말고 짧게 질문하거나 가정을 명시한다.

---

## 사용자 승인 필요 상황

다음 상황에서는 코드 수정 전에 사용자 확인을 받는다.

- 요구사항이 서로 충돌하는 경우
- 기존 동작을 크게 바꿔야 하는 경우
- 여러 구현 방식 중 선택이 필요한 경우
- 예상보다 수정 범위가 커지는 경우
- 새 의존성 또는 외부 라이브러리를 추가해야 하는 경우
- 데이터 삭제, 파일 이동, 설정 초기화 등 되돌리기 어려운 변경이 필요한 경우
- UI 배치나 동작이 요구사항과 다르게 해석될 수 있는 경우
- 성능 개선을 위해 기존 구조를 크게 변경해야 하는 경우

---

## 리소스 및 파일 처리 원칙

프로젝트에 이미지, 아이콘, 문서, 설정 파일 등 리소스가 있는 경우 다음 원칙을 따른다.

1. 기존 리소스 폴더 구조를 먼저 확인한다.
2. 임의로 새 리소스 경로를 만들기 전에 기존 경로를 우선 사용한다.
3. 파일명이 문서와 실제 폴더에서 다르면 실제 파일명을 확인하고 매핑한다.
4. 리소스가 누락되어 있으면 새 파일을 임의 생성하지 말고 필요한 파일명과 설명을 사용자에게 보고한다.
5. 경로 구분자, 대소문자, 공백, 한글 파일명 처리에 주의한다.
6. 리소스 변경이 필요한 경우 변경 전후 파일명을 보고한다.

---

## 문서 및 이미지 해석 원칙

사용자가 요구사항 문서나 이미지를 제공한 경우 다음 기준을 따른다.

1. 문서의 텍스트 요구사항을 우선 기준으로 삼는다.
2. 이미지는 UI 구조, 배치, 기능 구분, 예시를 이해하기 위한 보조 자료로 사용한다.
3. 이미지가 실제 구현 기준인지 참고용인지 문서에 명시되어 있으면 그 구분을 따른다.
4. 이미지와 텍스트가 충돌하면 충돌 내용을 사용자에게 보고하고 확인을 요청한다.
5. 참고용 이미지에 있는 모든 요소를 기능으로 추가하지 않는다.
6. “샘플”, “예시”, “참고”로 표시된 자료는 그대로 구현하지 말고 의도를 해석해 반영한다.

---

## 테스트 원칙

각 작업 후 가능한 범위에서 테스트 방법을 제시한다.

테스트 방법에는 다음을 포함할 수 있다.

- 실행 방법
- 확인해야 할 화면 또는 동작
- 기존 기능 회귀 테스트
- 오류 발생 시 확인할 로그 또는 파일
- 사용자가 직접 확인해야 할 항목

자동 테스트가 있는 프로젝트라면 기존 테스트 방식을 우선 사용한다.

자동 테스트가 없거나 실행이 어려운 경우 수동 테스트 체크리스트를 작성한다.

---

## 작업 재개 규칙

이전 작업이 중간에 멈췄거나 한도 문제로 중단된 경우에는 먼저 현재 상태를 분석한다.

보고할 내용:

- 이미 구현된 항목
- 아직 구현되지 않은 항목
- 일부만 구현된 항목
- 다음으로 진행해야 할 단계
- 중복 구현을 피하기 위해 주의할 부분

분석 후 사용자 승인을 받은 단계만 이어서 구현한다.

---

## 금지 사항

다음 작업은 사용자 승인 없이 하지 않는다.

- 전체 코드를 대규모로 재작성
- 기존 기능 삭제
- 설정 파일 초기화
- 데이터 파일 삭제 또는 이동
- 새 외부 의존성 추가
- 요구사항에 없는 기능 추가
- 테스트되지 않은 임시 코드를 최종 구현처럼 남기기
- 사용자가 승인하지 않은 단계까지 함께 구현

---

## Codex 작업 문서 위치 규칙

Codex 작업용 문서는 `.codex/` 폴더에 보관한다.

예:
- `.codex/EditProject.md`
- `.codex/PLANS.md`
- `.codex/FixBug.md`
- `.codex/ReviewCode.md`

새 작업 계획을 작성할 때는 프로젝트 루트가 아니라 `.codex/PLANS.md`에 작성한다.

단, `AGENTS.md`는 Codex가 프로젝트 규칙을 쉽게 인식할 수 있도록 프로젝트 루트에 둔다.
