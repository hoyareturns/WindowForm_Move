## 1. 전체 요구사항 요약

이번 개선의 목표는 Smart_Window의 적용 범위와 초기 표시 상태를 다시 정리하고,
마킹 기능을 별도의 이동 가능한 도구 창과 모니터별 작업 구조로 재구성하는 것이다.

주요 요구사항은 다음과 같다.

- Smart_Window은 최대화 또는 화면 절반 배치 창에만 표시한다.
- 프로그램 실행 시 각 Smart_Window 툴바는 접힌 상태로 시작한다.
- 창 위치, 프로그램 실행 등 각 버튼 세트의 기본 펼침 상태를 설정할 수 있게 한다.
- 모든 버튼의 표시 여부, 이름, 단축키를 설정에서 관리한다.
- 필수 버튼은 숨길 수 없게 한다.
- 메모 입력창의 Enter는 확인이 아니라 줄바꿈으로 동작하게 한다.
- 캡처 저장 및 클립보드 이미지의 해상도를 개선한다.
- 기존 마킹 버튼들을 별도의 이동 가능한 마킹 도구 창으로 옮긴다.
- 마킹 대상 모니터 또는 직접 선택 영역을 지정할 수 있게 한다.
- 마킹 중에도 다른 모니터와 다른 프로그램을 사용할 수 있게 한다.
- 신규 도형 도구와 상태 표시, 폴더 열기, 명시적 마킹 종료 기능을 추가한다.
- 캡처 후에도 마킹 상태를 유지하고 저장 폴더를 자동으로 열지 않는다.

요구사항 문서의 첫 번째 이미지는 실제 기능 구성 기준으로 사용한다.
두 번째 큰 이미지는 그룹 구성, 간격, 상태 표시 등 디자인 참고용으로만 사용한다.

## 2. 현재 상태와 주의 사항

- 현재 작업 트리에는 아직 커밋되지 않은 변경이 존재한다.
- 미커밋 변경에는 자동 시작, Smart_Window 접기 버튼, 일반 창 및 대화상자까지
  적용 범위를 넓힌 코드가 포함되어 있다.
- 최신 요구사항은 일반 창 적용을 다시 제외하므로 해당 변경을 그대로 확장하지 않고
  요구사항 기준으로 정리해야 한다.
- 기존 마킹은 전체 가상 화면을 캡처한 뒤 전체 화면을 정지시키는 구조다.
- 모니터별 마킹과 다른 창 동시 작업은 현재 구조의 부분 수정만으로 해결하기 어렵다.
  마킹 세션, 렌더링 폼, 입력 처리의 책임을 분리하는 단계적 구조 개선이 필요하다.
- 기존 설정 파일 경로와 JSON 데이터는 유지하며 새 설정은 기본값을 제공해야 한다.
- 준비된 신규 아이콘은 `Assets/Icons`의 실제 파일명을 우선 사용한다.

### 1단계: 기준 동작 정리와 툴바 초기 상태

#### 목표

#### 작업 범위

- 새 OverlayForm은 접힌 상태로 시작

- 기존 자동 시작 옵션은 유지

## 4. 제외 범위

이번 계획에서는 다음 작업을 포함하지 않는다.

- 다른 프로그램 프로세스에 DLL을 주입하여 제목 표시줄에 실제 버튼을 삽입하는 방식
- 기존 JSON 설정 파일 초기화 또는 저장 데이터 삭제
- 외부 UI 프레임워크 또는 신규 NuGet 패키지 도입
- 요구사항 이미지의 복사, 붙여넣기, 크기 변경, 회전 등 참고용 기능의 임의 추가
- 기존 창 위치 저장 및 프로그램 실행 기능의 동작 변경
- 사용 설명서와 PPT 최종 제작
- 저장소 이름 또는 GitHub 저장소 이름 변경

## 5. 공통 테스트 전략

- 각 단계 완료 시 `dotnet build -c Release`
- 경고 및 오류 0개 확인
- `win-x64`, self-contained, single-file publish 확인
- `publish-single/Smart_Window.exe` 교체 확인
- 1개 및 3개 모니터 환경 수동 테스트
- 좌우 및 상하 모니터 배치 테스트
- Excel, Outlook, 브라우저, 탐색기, 일반 대화상자 회귀 테스트
- 100%, 125%, 150% DPI 환경 테스트
- 기존 설정 파일을 유지한 업그레이드 테스트
- 새 사용자 환경처럼 설정 폴더가 없는 상태의 초기 실행 테스트

## 6. 사용자 확인이 필요한 사항

구현 전에 다음 항목을 확인해야 한다.

1. 최신 요구사항을 기준으로 일반 크기 창과 대화상자에서는 Smart_Window을 완전히
   숨기는 것이 맞는지 확인이 필요하다. 현재 미커밋 구현은 해당 창에도 접기 버튼을
   표시하도록 되어 있어 요구사항과 충돌한다.
2. “모든 버튼 표시 여부/이름/단축키 설정”의 범위가 메인 Smart_Window 버튼뿐 아니라
   별도 마킹 도구 창의 모든 버튼까지 포함하는지 확인이 필요하다.
3. 단축키는 시스템 전체에서 동작하는 전역 단축키인지, Smart_Window 또는 대상 창이
   활성화된 경우에만 동작하는 단축키인지 선택이 필요하다.
4. 마킹 대상 모니터를 변경할 때 기존 마킹을 유지하여 새 대상과 함께 관리할지,
   대상별로 별도 보관할지 확인이 필요하다.
5. 직접 영역 선택은 선택한 고정 화면을 캡처해 마킹하는 방식인지, 선택 영역 아래의
   프로그램이 계속 움직이는 실시간 방식인지 확인이 필요하다.
6. 마킹 도구 플로팅 창의 마지막 위치를 재실행 후에도 저장할지 확인이 필요하다.

## 7. 권장 구현 순서

사용자 승인 후 한 번에 전체를 구현하지 않고 다음 순서로 진행한다.

1. 1단계 기준 동작 정리
2. 2단계 독립 오류 수정
3. 3단계 마킹 도구 플로팅 창
4. 4단계 모니터별 비차단 마킹
5. 5단계 신규 도형
6. 6단계 캡처 상태 유지
7. 7단계 버튼 및 단축키 설정

각 단계가 완료될 때마다 실행 파일과 수동 테스트 결과를 확인하고 다음 단계 승인을 받는다.

## 후속 수정: 툴바 Z-order 및 단일 실행 안정화

### 요구사항 요약

- 최대화 창에만 Smart_Window 툴바를 표시한다.
- 각 모니터의 대표 최대화 창에 툴바 하나를 유지한다.
- 툴바는 대상 창보다 위에 표시하되, 대상 창을 덮은 다른 프로그램보다 위로 올라오지 않는다.
- 대상 창을 클릭해 활성화해도 툴바가 사라지지 않아야 한다.
- Smart_Window를 다시 실행하면 기존 Smart_Window 프로세스를 종료하고 새 실행만 유지한다.

### 구현 범위

- 외부 대상 창을 Win32 소유 창으로 지정하는 방식을 제거한다.
- 툴바를 대상 창 바로 위의 Z-order에 비활성 상태로 배치한다.
- 시작 시 같은 이름의 기존 Smart_Window 프로세스를 종료하고 종료 완료 후 새 인스턴스를 시작한다.
- 모니터별 최대화 창 선택과 작업 영역 우측 상단 배치는 기존 기준을 유지한다.

### 제외 범위

- 절반 창 및 일반 크기 창의 툴바 지원
- 마킹, 캡처, 창 이동 기능 변경
- 설정 파일 초기화 또는 기존 사용자 데이터 변경

### 예상 수정 파일

- `Program.cs`
- `OverlayForm.cs`
- `WindowController.cs`
- 필요 시 단일 실행 처리를 위한 소규모 헬퍼 파일
